# Day 004. 로그아웃 처리와 쿠키/세션의 상관관계

## 개념 노트: 쿠키와 세션은 무슨 사이일까?
- 세션(Session): 서버의 메모리에 저장되는 원본 장부. 보안이 매우 뛰어나지만, 사용자가 많아지면 서버 메모리가 무거워진다는 단점이 있음
- 쿠키(Cookie): 클라이언트의 임시 저장소에 보관됨. 브라우저는 서버에 요청을 보낼 때마다 이 쿠키를 헤더에 담아서 보냄. 쿠키 자체는 변조되거나 탈취될 위험이 있어서, 절대 쿠키 안에 "비밀번호"나 "권한"같은 중요 정보를 직접 적어두면 안됨. 의미 없는 Session ID만 적어두는 것이 좋음

## 로그아웃의 본질
- 서버 메모리에서 이 사용자의 세션 데이터를 완전히 파기해버리는 것
- 장부에서 지워지면 해커가 훔쳐도 서버에 들어올 수 없음

## MemberController에 로그아웃 API 추가하기
로그아웃 로직은 비즈니스 로직(DB 조회 등)이 전혀 필요 없고, 오직 HTTP 세션만 제어하면 되기 때문에 Service 계층을 거칠 필요 없이 Controller에서 바로 처리하는 것이 깔끔함
```java
    @PostMapping("/logout")
    public ResponseEntity<ApiResponse<Void>> logout(
        HttpServletRequest request
    ) {
        // 현재 요청에 포함된 세션을 가져옴
        // getSession(false): 기존 세션이 있으면 가져오고, 없으면 새로 만들지 말고 null
        HttpSession session = request.getSession(false);

        if (session != null) {
            session.invalidate(); // 서버 메모리에서 세션 장부 파기
        }

        return ResponseEntity.ok(ApiResponse.success(null));
    }
```

## Session에 대한 나름의 이해?
그냥 하나의 거대한 Session을 관리하는 Map이 있다고 생각하자.
그렇다면 Session<ID, HttpSession>의 형태라고 생각을 해보자.
```JavaScript
// 🏢 톰캣(서버)의 거대한 세션 저장소 (Global Map)
{
    // 👦 A유저의 JSESSIONID
    "JSESSIONID=AAA": {  // <--- 이 중괄호 하나가 바로 HttpSession (개인 사물함, 또 다른 Map)
        "LOGIN_MEMBER": 1,        // 우리가 setAttribute로 넣은 값!
        "USER_ROLE": "USER",      // 필요하면 권한도 넣을 수 있음
        "LAST_ACCESSED_TIME": "14:30:00" // 스프링이 알아서 관리하는 접속 시간
    },

    // 👧 B유저의 JSESSIONID
    "JSESSIONID=BBB": {  // <--- B유저의 HttpSession
        "LOGIN_MEMBER": 2
    }
}
```
- 대충 이런 형태라고 할 수 있음
- JSESSIONID=AAA 이 쿠키를 이걸 헤더에 보내서 HttpServletRequest로 보내는 것임
